# コンプライアンス・ナレッジベース検索（週次課題プロトタイプ）

条文IDで紐づけた法令データを、キーワードで検索できるWebアプリケーションです。
PHP / HTML / CSS / JavaScript / MySQL の週次課題として作成したものですが、将来的な事業構想
「AI活用による不動産投資運用の自動化支援」における MVP（公開情報を条文IDで紐づけたナレッジベース）
の段階的な実装としても位置づけています。

---

## できること

1. キーワード（またはタグ）を入力すると、DBに登録された条文の中から関連するものを検索する
2. ヒットした条文のうち1件について、デジタル庁が提供する公式の **e-Gov法令API** から
   正式な条文原文をリアルタイムで取得し、あわせて表示する
3. e-Govの該当条文ページへ直接飛べるリンクを表示する
4. 管理画面から条文データをブラウザ上で登録・修正・削除できる（CRUD全対応）

対象データは「不動産・宅建」（宅地建物取引業法）と「金融・コンプライアンス」（金融商品取引法）の
2テーマ、計8条文を初期データとして収録しています。管理画面から随時追加できます。

## 動かし方

### 事前準備：DBとテーブルの作成

1. `http://localhost/phpmyadmin/` を開く
2. DB名 `gs_php_db` を作成する
3. テーブル名 `laws_table` を作成し、以下のカラムを設定する

| カラム名 | データ型 | 長さ | その他 |
|---|---|---|---|
| `id` | INT | 11 | PRIMARY / A_I |
| `law_name` | VARCHAR | 128 | |
| `law_id` | VARCHAR | 64 | |
| `article_label` | VARCHAR | 32 | |
| `article_num` | VARCHAR | 16 | |
| `category` | VARCHAR | 64 | |
| `tags` | VARCHAR | 256 | |
| `summary` | TEXT | - | |

4. phpMyAdminのSQLタブで `data/insert_initial_data.sql` の内容を実行し、初期データを投入する

### ローカル（XAMPP / MAMP）

1. `htdocs` 配下にフォルダ一式をそのままコピーする
2. Apache（とMySQL）を起動する
3. ブラウザで `http://localhost/（フォルダ名）/` を開く

### さくらインターネットのレンタルサーバー

1. FTP等でフォルダ構造を保ったままアップロードする
2. `db_connect.php` の接続情報（`$dbn` / `$user` / `$pwd`）をサーバーのDB設定に合わせて変更する
3. ブラウザで設置先のURLにアクセスする

`search.php` は `file_get_contents` が使えない環境（`allow_url_fopen = Off`）でも、
自動的に `cURL` にフォールバックして e-Gov法令API を呼び出すため、サーバー環境を選ばず動作します。

---

## 検索の仕組み（中身）

1. `app.js` がキーワードを `search.php?keyword=...` へ非同期（fetch）で送る
2. `search.php` がDBの `laws_table` に対し、`summary` / `tags` / `article_label` / `law_name`
   を対象にSELECT文（LIKE句による部分一致）を実行する
3. ヒットした最初の1件について、`law_id` と `article_num` を使って e-Gov法令API
   （`https://laws.e-gov.go.jp/api/1/articles;lawId=...;article=...`）にリクエストし、
   返ってきたXMLから条文本文（`Sentence`要素）を抽出する
4. 結果をJSONで `app.js` に返し、画面に描画する

---

## データの追加・編集・削除方法

ブラウザから `admin_input.php` を開き、フォームに入力して登録する。
`admin_select.php` の一覧画面から各行の「編集」「削除」ボタンで修正・削除できる。

法令IDは `https://laws.e-gov.go.jp/` で法令を検索し、URLの末尾から確認する。

## 週次課題の取組内容・工夫・気づき

### 週1（PHP・HTML・CSS・JavaScript・外部API）

**取組内容**

- HTML / CSS / JavaScript / PHP を役割ごとに別ファイルに分割して構成した
- 条文をキーワードで検索し、ヒットした条文についてe-Gov法令API（デジタル庁公式・無料・認証不要）
  から正式な条文原文をリアルタイムで取得・表示する仕組みを実装した
- 法令データはJSONファイルで管理し、タグ・要約・条文IDを手動で整理した

**工夫した点**

- `file_get_contents` が使えないサーバー環境でも `cURL` に自動フォールバックするよう実装し、
  ローカル（XAMPP）とさくらレンタルサーバーの両方で動作するようにした
- HTMLファイルを`file://`で直接開く方式とApache経由の`http://localhost/`の違いを
  理解した上で、後者でしかPHPが動かない理由を意識した設計にした

**難所・苦労した点**

- ファイルに`AIAM_proto_`という接頭辞をつけてリネームした際に、コード内の参照（`href`・`src`・
  `fetch`先のURL）も全部変えなければならず、404エラーで動かなくなった
- VS CodeのLive Server拡張（`127.0.0.1:5500`）でHTMLを開いており、XAMPPのApache経由でない
  ために`file://`プロトコルの制限でCSSとJSが読み込めない問題に気づくまでに時間がかかった
- Gitのコミットメッセージで全角引用符（`"`）を使ったためコマンドが入力待ちになり止まった
  （`Ctrl+C`でキャンセルし、半角で再入力して解決）

**気づき・学び**

- 「ファイルをダブルクリックで開く」と「Apacheサーバー経由でブラウザに返す」は、
  見た目は似ているが全く別の仕組みであることを体感した
- ファイル名を変えるときは「呼ぶ側のコード」も必ずセットで変える必要があること
- コマンドラインでの記号（全角・半角）の違いが動作に影響すること

---

### 週2（MySQL・SQL・PHP×DB連携）

**取組内容**

- phpMyAdminでDB（`gs_php_db`）とテーブル（`laws_table`）を作成し、カラム設計を行った
- 週1でJSONファイルで管理していた条文データをMySQLのテーブルに移行した
- 条文データをブラウザから登録・一覧表示・修正・削除できる管理機能（CRUD全操作）を実装した
- DB接続設定を `db_connect.php` として共通ファイルに切り出し、各PHPから `require_once` で読み込む形にした
- 初期データを一括投入できるSQLファイル（`data/insert_initial_data.sql`）を別途用意した

**工夫した点**

- ユーザー入力をSQLに直接埋め込まず、バインド変数（`bindValue`）を使って
  SQLインジェクションを防いだ
- 削除ボタンには `confirm()` による確認ダイアログを挟み、誤操作でデータが消えないようにした
- DB接続設定を共通ファイル（`db_connect.php`）に集約したことで、接続先が変わった際の
  修正箇所を1ファイルだけに絞れる構造にした
- 週1の検索画面（`index.html` / `app.js`）はそのまま残し、バックエンドの`search.php`だけを
  JSON参照からDB参照に切り替えることで、既存のUIを壊さずにデータ層を移行した

**難所・苦労した点**

- 手入力でデータを登録する際、法令IDを都度e-Gov法令検索のURLから確認する必要があった
  （対策として、初期8件を一括投入するSQLファイルを別途作成した）
- 登録フォームで入力したデータに誤りがあった際、UPDATE/DELETE機能がまだ実装されておらず
  修正・削除ができない状態が一時的に発生した
  （その場でUPDATE・DELETE機能を追加実装して対応した）

**気づき・学び**

- PHPファイルを上書きしてもDBのデータは消えない、という「プログラム」と「データ」の
  分離を体感した。週1のJSONファイル方式ではファイルにデータが直接書かれていたため、
  この分離が今回初めて実感できた
- テーブル設計（カラム名・型・制約）をアプリ稼働前に決めておく重要性を学んだ
- SQLの`WHERE id = :id`のようにIDで1件だけを対象にする書き方と、
  WHERE句を書き忘れると全件が更新・削除される危険性を意識するようになった

## 現時点でのスコープと、今後の発展方向

これは**ルールベースのキーワード検索**であり、AIエージェントではありません。
入力されたキーワードと、登録済みのタグ・要約文が一致するかどうかだけを見ています。

将来的に発展させる場合の方向性（事業構想のロードマップに対応）：

- **Step1（週1・週2）**：公開法令を条文IDで紐づけたナレッジベースの検索基盤
  （週1：JSONで試作 → 週2：MySQLに移行、CRUD完備）
- **Step2**：自社のDD・契約書データとのDB上での紐づけ
- **Step3**：DD文書の文脈を読み取り、関連条文を自律的に判断して提示する
  （AIエージェント的な振る舞いへの拡張）

なお、現時点では対象法令を宅地建物取引業法・金融商品取引法の一部条文に絞っている。
e-Gov法令APIには各法令の全条文を取得する手段があり、技術的には全条文対応も可能だが、
全条文の網羅よりも「どの条文が今の状況（DD・契約書の文脈）に関連するか」を整理することの方が
本質的な価値であり、タグ付け方針の精緻化は自社データと統合する段階（Step2以降）で
体系的に検討する方が適切と判断し、今回のスコープからは意図的に外している。

## 利用データについて

法令データは e-Gov法令検索（デジタル庁）が提供する公開情報を参照しています。
e-Govのデータは「政府標準利用規約（第2.0版）」に基づいて利用可能です。
本READMEおよびコード中のサンプルデータは学習・課題提出目的での利用を想定しています。
